# DDA — 数据库死锁检测与 LLM 决策系统

## 项目背景

rookieDB 是一个教学级关系数据库，实现了完整的多粒度锁机制（S/X/IS/IX/SIX）和事务管理，但**没有死锁检测**。当多个事务以相反顺序获取锁时，形成循环等待（deadlock），事务永久挂起。

所有主流数据库的 deadlock victim selection 全部使用固定规则：

| 数据库 | Victim 选择规则 |
|--------|---------------|
| MySQL (InnoDB) | 回滚 undo log 最小的（插入/更新/删除行数最少），回滚代价最低 |
| PostgreSQL | 回滚触发环闭合的那个事务（最后发起锁请求导致死锁的那个） |
| CockroachDB | 回滚优先级最低的；优先级相同回滚最年轻的 |
| Oracle | 回滚导致死锁的那个事务（谁触发环闭合杀谁） |
| SQL Server | 回滚 `DEADLOCK_PRIORITY` 最低的；优先级相同回滚代价最小的 |

**所有规则都是固定的。被选中的事务只知道"我被回滚了"，不知道为什么。**

DDA 验证一个命题：**能否用 LLM 替代数据库内核中的固定规则，让 victim selection 有上下文分析、有语义判断、有自然语言解释？**

---

## 核心设计决策

> 以下决策过程记录在设计讨论文档中。这里是结论——每个决定背后的 engineering judgment。

| 决策 | 结论 | 理由 |
|------|------|------|
| 不用 Worker Agent | 砍掉 | 固定 SQL 不需要 LLM 执行。并发事务用纯代码，DDA 是唯一用到 LLM 的地方 |
| 不拆分 DDA 为多 Agent | 单 DDA 够用 | 2 事务 1 个环的场景撑不起 Hand-off 模式的复杂度。拆分思路留作扩展 |
| 不加 Orchestrator | 不加 | 无动态任务需要编排 |
| 直接上 LLM | 纳入 Phase 1 | 纯代码规则数据库已做了几十年，没有增量价值。LLM + 规则 fallback 共存 |

## 实施路线

### 阶段一：传统算法 + 对比基线

实现完整的死锁检测与恢复链路——wait-for graph + DFS 找环 + **固定规则选 victim**。

同时实现三种主流数据库的固定规则，同一死锁场景下可切换对比：

| 策略 | 规则 | 对应数据库 |
|------|------|-----------|
| Min Locks | 回滚持有锁数量最少的事务 | 类似 MySQL |
| Youngest First | 回滚最后开始的事务 | 类似 CockroachDB |
| Cycle Trigger | 回滚触发环闭合的事务 | 类似 PostgreSQL/Oracle |

**产出**：三种策略在同一场景下的对比数据——各选了哪个 victim、为什么。

### 阶段二：LLM Victim Selection

将固定规则替换为 LLM 决策。同一死锁场景下，对比 LLM 和固定规则的决策差异。

LLM 接收死锁环上各事务的锁状态描述，分析后选定 victim 并给出自然语言解释。LLM 分析失败时，用阶段一的规则兜底。

**产出**：LLM 决策与传统规则的对比——选得可以相同可以不同，但 LLM 每次都有可理解的理由。

## 功能需求

### 1. 锁状态查询

获取数据库中所有活跃事务的锁持有和等待关系。查询失败时不影响系统继续运行。

### 2. 死锁检测

从锁状态数据中判断是否存在死锁（事务间循环等待）。检测是确定性的，不依赖 LLM。输出死锁环上涉及的所有事务。

### 3. Victim Selection

**阶段一**：使用固定规则（Min Locks / Youngest First / Cycle Trigger）选定回滚目标，支持策略切换。

**阶段二**：LLM 分析候选事务的持有锁、等待链位置和回滚代价，选定 victim 并给出自然语言解释。LLM 失败时规则兜底。

### 4. 死锁解除

对被选中的事务执行回滚，解除死锁。回滚通过独立连接执行。

### 5. 恢复验证

验证回滚后非 victim 事务能继续执行并成功提交；被回滚的事务连接正确收到错误。

### 6. 持续监控

系统在并发事务运行期间持续监控锁状态，不阻塞事务的正常执行。

## 验收标准

**阶段一**：
- 死锁发生后 3 个轮询周期内检测到（默认 500ms 间隔，即 ≤1.5s）
- 选定 victim 后 1 个周期内完成 ROLLBACK
- 三种固定规则可在同一场景切换，输出不同 victim 和选择理由
- 回滚后非 victim 事务在 5s 内继续执行并成功提交
- 被回滚事务的连接收到正确的错误信息

**阶段二**：
- LLM 选定的 victim 及理由与锁状态事实一致（无幻觉）
- LLM 调用延迟不超过 3s（同场景快速对比）
- 同一场景下，LLM 与固定规则形成可用表格对比的输出
- LLM 调用失败时，规则 fallback 在 1 个周期内接管

**通用**：
- 单命令 `python dda_basic.py` 启动，全程无需手动干预
- 全过程可观测：终端实时输出轮询状态、图结构、victim 选择理由、回滚结果

## 技术约束

| 约束 | 说明 |
|------|------|
| 语言 | Python 3 |
| LLM 调用 | Anthropic SDK（仅用于 victim selection） |
| 数据库 | rookieDB（本地单实例，多粒度锁） |
| 通信 | TCP socket (localhost:18600) |
| 并发模型 | asyncio。I/O 密集型：TCP socket 轮询 + LLM API 调用，async/await 天然适配；单事件循环管理并发事务和 DDA，避免多线程同步开销 |
| 依赖 | 标准库 + Anthropic SDK，无 LangChain/CrewAI |

## 深挖路线图

> 基础版完成后，按以下路线继续深挖。三个层级按优先级排列。

### 第一层：场景与评估（不需改数据库）

- **多死锁场景 Benchmark**：三事务环、共享锁死锁、多环并发等场景，每个场景跑三种固定规则 + LLM，出对比评估表格
- **DDA 抽象化**：`BaseDeadlockDetector` + `BaseVictimSelector` + 策略模式，可扩展架构
- **检测延迟分析**：统计各场景死锁从发生到检测的延迟，数据支撑工程讨论

### 第二层：rookieDB 增强（改数据库）

- **事务元信息**：在 LockManager 中增加事务开始时间、受影响行数、SQL 文本追踪，让 LLM 有更丰富的决策上下文
- **死锁预警**：不等环闭合，在等待链增长时提前预警——传统数据库没有的能力
- **锁升级协商**：Agent 间协商锁资源，探索 AI Agent 间的资源管理

### 第三层：接真实数据库

- **PostgreSQL 适配**：改 `pg_locks` + `pg_stat_activity` 为数据源，DDA 独立于 Pg 运行
- **对比评估**：Pg 内置死锁检测 vs DDA LLM 决策，同一场景对比
- **多数据库支持**：rookieDB / PostgreSQL / MySQL 三种数据源

### 推荐顺序

```
基础版 → 多场景 Benchmark → rookieDB 事务元信息 → DDA 抽象化 → PostgreSQL 适配
```
